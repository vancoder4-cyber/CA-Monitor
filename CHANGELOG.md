## 2026-07-14 · 「确认」支持指定日期(修一个会确认错的真 bug)

- **Bug**:`确认 KLAC 2.3` 不带日期时会匹配该标的的**任意**事件。而 KLAC 有两条冲突、正确值不同(05-18 是 `2.3`、02-17 是 `1.9`),ASML 同理(`3.16845` / `1.9064`)—— 一次确认会把错的值盖到另一条上。
- **修**:`确认 代码 [正确值] [日期]`。先摘 `YYYY-MM-DD`(否则 `2026` 会被当成金额),再取数值;按日期在 conflicts/pending/calendar/gaps 里定位事件类型。不带日期时行为不变(取该标的第一条冲突)。
- 同一标的有多条**值不同**的异常时,**必须带日期**,如 `确认 KLAC 2.3 2026-05-18`。
- 四处同步:`COMMANDS` desc / README 指令清单 / `confirm_card` 用法文案 / 关于卡片。

## 2026-07-14 · 门禁补漏:日历也纳入 + 单源金额同样封锁
- **发现日历视图是另一套取值代码**(`run.py._ck` / `report._pick` 都是「第一个源赢」),**完全绕过了多数票、确认覆盖和门禁** —— 上一版只堵了「待执行/催办/新公告」
- 取值逻辑收敛为唯一真相 `reconcile.pick_value()`,run.py 与 report.py 全部改走它;日历格子也上门禁(冲突→`⚠️待确认`,单源→`⚠️单源`)
- **单源金额同样封锁**:只有 1 个源报的金额/比例 = 没交叉验证过,不能当权威值。三面统一显示
  **「⚠️单源未交叉验证(x)· 待人工确认,勿据此执行」**
  - 实例:TSM 9/16 只有 Alpaca 一个源(报的还是 ADR 扣税净额),多数票无从投起 —— 现在直接封锁而不是印一个数
  - 实例:CRWD 4:1、KLAC 10:1 的拆股比例也是单源 → 同样需人工确认
- 人工确认后 `g.acked=True`,门禁解除、按确认值显示(日历/详情列/卡片全生效)
- **人工确认改为对「所有事件组」生效**(原来只作用于冲突):单源事件也能被人工放行,否则确认了门禁还封着
- 实测:KLAC 10:1 单源拆股 → 封锁 → 发「确认 KLAC」→ 放行显示 `10:1`

## 2026-07-14 · 金额门禁:未确认冲突的事件不给「确定值」
- 之前即使各源对不上,卡片/网页照样印一个金额,运营可能直接照着执行 —— 这是真正会绕过人工的地方
- 现在:只要该事件还有**未确认的冲突**,三个面一律不显示确定金额,改为
